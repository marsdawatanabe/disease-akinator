# 病気アキネーター 要件定義書 v3

## プロダクト概要
アキネーター風のシンプルなYes/No質問に答えるだけで、可能性のある病気を関連度付きで提示し、受診すべき科を案内するWebアプリ。

## コンセプト
- 「聞かれてないから言わなかった」症状をAIの質問で引き出す
- 二択で答えるだけだからユーザーの負荷が低い
- 「え、それも関係あるの？」という意外な質問の意図が見える

## ターゲットユーザー
- 体調不良だが何科に行けばいいかわからない人
- 症状はあるが「わざわざ病院に行くほどか」迷っている人
- 問診前に自分の症状を整理したい人

## 競合との差別化（vs ユビー等）
- 質問が少ない（10-15問 vs ユビー20-30問）
- 結果がわかりやすい（「まず○科へ」の一本化）
- 「なぜこの質問？」が見える（各質問の医学的意図を説明）

## ユーザーフロー

### ① 基本情報入力
- 年齢（選択式: 0-10, 11-20, 21-30, 31-40, 41-50, 51-60, 61+）
- 性別（男性/女性/回答しない）
- 目的: ベイジアン推論の事前確率に反映

### ② 緊急チェック（チェックリスト一括表示）
以下に1つでも該当 → 即「今すぐ119に電話 / #7119に相談」画面へ遷移。質問フローに進まない。

- □ 激しい胸の痛み・圧迫感
- □ 呼吸が非常に苦しい
- □ 意識がもうろうとしている
- □ 体の片側が動かない・しびれる
- □ 突然の激しい頭痛（今まで経験したことがないレベル）
- □ 大量の出血が止まらない
- □ 上記に該当しない → 質問フローへ

### ③ 通常質問フロー
- 最大10-15問（情報エントロピーベースで動的に次の質問を選択）
- 回答選択肢: はい / いいえ / わからない
- 各質問に「なぜこの質問？」展開ボタン（タップで説明表示、デフォルト非表示）
- プログレスバー表示（あと何問程度か）
- 「わからない」が半数以上 → 早期終了してフォールバック画面へ

### ④ 結果画面
- 関連度の高い疾患候補（上位3-5件）を関連度順で表示
- 各候補に「なぜ関連するか」の説明（どの回答が影響したか）
- **「まず○科を受診してください」**（受診科を1つに絞って提示）
- 候補が複数科にまたがる場合は最も関連度の高い疾患の科を提示
- 免責表示（強く、目立つデザインで）

### ⑤ フォールバック画面（情報不足時）
- 「お答えいただいた情報では十分な判断ができませんでした」
- 「まず内科（かかりつけ医）を受診してください」
- 免責表示

## 免責表示ポリシー
- **開始画面**: 「このサービスは医療診断ではありません。参考情報としてご利用ください。体調に不安がある場合は必ず医療機関を受診してください。」
- **結果画面**: 上記 + 「表示される情報はAIによる推定であり、医師の診断に代わるものではありません。」
- **デザイン**: 背景色付きバナー、スキップ不可

## 技術要件

### スタック
- フロントエンド: Next.js (App Router) + TypeScript + Tailwind CSS
- 推論エンジン: クライアントサイドのベイジアン推論（サーバー不要）
- 疾患DB: JSONファイル（初期50-100疾患）
- デプロイ: Vercel（静的サイト寄り、無料枠）

### ベイジアン推論エンジン
- 各疾患の事後確率をベイズの定理で逐次更新
- 質問選択: 情報エントロピー最大化（最も情報利得の高い質問を動的に選択）
- 年齢・性別による事前確率の調整
- 「わからない」回答時は事後確率を変更しない（その症状について情報なしとして扱う）
- 収束条件: トップ候補の関連度が十分高い or 質問数上限到達

### 疾患DBスキーマ
```json
{
  "diseases": [
    {
      "id": "string",
      "name": "疾患名",
      "department": "受診科",
      "description": "疾患の簡単な説明",
      "baseProbability": { "male": 0.0-1.0, "female": 0.0-1.0 },
      "ageModifier": { "0-10": 1.0, "11-20": 1.0, ... },
      "symptoms": [
        {
          "id": "symptom_id",
          "question": "質問文",
          "weight": 0.0-1.0,
          "explanation": "なぜこの質問をするかの説明"
        }
      ]
    }
  ],
  "emergencyChecks": [
    {
      "id": "string",
      "label": "チェック項目テキスト",
      "action": "119に電話 / #7119に相談"
    }
  ]
}
```

### 疾患DB構築方針
- AI（Gemini/Claude）に医学文献ベースで生成
- 公知の医学情報（症状→疾患の対応関係）を活用
- 生成後に人間がレビューして明らかな誤りを除去
- 初期は一般的な内科・耳鼻科・皮膚科・整形外科の疾患を中心に50-100件

## スコープ外（v1ではやらない）
- ユーザー登録・ログイン
- 問診結果の保存・共有
- 医師とのチャット連携
- 多言語対応（日本語のみ）
- 高度なアクセシビリティ対応
- マネタイズ

---

## リスクと対策
| リスク | 対策 |
|--------|------|
| 法的（医師法抵触） | 「関連度」表現、強い免責表示、診断行為ではなく情報提供と位置づけ |
| 精度（DBの品質） | AI生成+人間レビュー、初期は一般的疾患に限定 |
| 倫理（重篤疾患の見落とし） | 緊急チェック必須化、フォールバック画面で受診推奨 |
| 技術（質問数で精度出るか） | 情報エントロピーベースの質問選択、リリース後にチューニング |

## 成功指標（v1）
- 質問数: 平均10問以内で結果表示
- ユーザー完了率: 80%以上（途中離脱20%以下）
- 結果の納得感: ユーザーテストで「なるほど」と思える結果が出る
